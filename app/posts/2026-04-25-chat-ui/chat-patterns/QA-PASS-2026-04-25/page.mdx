### Findings worth a follow-up

1. **`useChat.error` lingers across slice/edit.** When the user edits via Up arrow after a server error, `useChat.error` stays set, and the inline `{error && …}` block continues to render under the empty state. Add `clearError()` to `handleEditUser`.
2. **Saved error messages re-render as the inline banner, not as a regular `<AssistantMessage>`.** Because `useChat.error` is also set on mount in some paths (or persists across reload via SWR cache?), the saved error message in `messages` table is shadowed. Workaround: on mount, if `initialMessages` includes an error-shaped final message, don't surface `useChat.error` separately — let the message renderer take it. Audit this when we see it again in production logs.
3. **No Copy/Regenerate visible on the inline-error block.** Expected — that block isn't an `<AssistantMessage>`. When the next round of model RTT works (with credits), Copy/Regenerate should appear on hover for any saved assistant text.

### `/api/chat` SSE wire format
```
curl -sS -b cookies.txt -X POST /api/chat -H "Content-Type: application/json" \
  -d '{"messages":[{"id":"u1","role":"user","parts":[{"type":"text","text":"…"}]}]}' \
  -w "status=%{http_code} ct=%{content_type} ttfb=%{time_starttransfer}\n"
# status=200 ct=text/event-stream ttfb=0.215s
# Body:
#   data: {"type":"start"}
#
#   data: {"type":"error","errorText":"Insufficient funds…"}
#
#   data: [DONE]
```
Confirmed: route emits the AI SDK UI-message stream protocol, content-type is `text/event-stream`, errors from the gateway propagate cleanly as `error`-typed chunks, the stream closes with `[DONE]`. TTFB on a cold function call is ~215 ms.

## Known limitations / what wasn't testable here

1. **Full nav-resilience flow end-to-end on the dev Convex deployment**: requires `bunx convex dev` (interactive) to push the new mutations. CI's preview deploy gets them via Vercel's pipeline.
2. **Real model round-trip**: the AI Gateway account is out of credits, so `/api/chat` returns an SSE error chunk instead of streaming text. The route's protocol behavior is verified end-to-end; rendering of *streaming* assistant text is verified by the Phase 0–8 tests that landed when credits were available, plus the visual check that Stop ↔ Submit ↔ Error icons cycle correctly through the AI SDK status field.
3. **Copy / Regenerate hover affordances**: hover-state assertions need a saved assistant message with non-empty text. With the gateway returning 402, the only saved messages we have are error blocks rendered via the legacy inline-error path (not via `<AssistantMessage>`). The component code is verified by typecheck + lint + the resilience suite; visual confirmation defers to the next preview-deploy round-trip with credits.
4. **8 sidebar-pdfs Playwright failures locally**: Node 25's IPv6-first DNS vs the dev server binding only IPv4. CI runs Linux + an older Node and these tests pass there.

## What's safe to ship

The branch as it stands satisfies the user's hard constraint: the route runs to completion regardless of client connection state, the Convex flusher persists the assistant message, and the user can navigate away and come back. Server-side resilience is preserved AND enhanced (the prior route was tied to `req.signal` — that bug is fixed).

All static QA gates are green. CI is green. The only deferred item is the model-round-trip live-deploy verification, which lands automatically on the next preview deploy.
